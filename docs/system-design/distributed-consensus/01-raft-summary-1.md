# Raft算法总结（一）

# 复制状态机

共识算法通常是基于复制状态机（Replicated State Machine, RSM）模型。根据这种模型，可以在一系列服务器上根据相同状态的状态机计算出相同的结果副本，即使在一些服务器宕机是也可以继续操作。复制状态机被用来解决分布式系统中各式各样的容错问题。

## 抽象定义

状态机是由一组描述状态的变量和一组修改变量的命令组成的软件抽象，它的状态只取决于它的初始状态和修改这些状态的命令序列。复制状态机是这样一种抽象，它可以使得多个状态机以相同的顺序执行相同的命令，输出相同的结果，具有相同的状态变化过程，达到相同的最终状态。

复制状态机抽象定义了两个接口事件，一个是输入事件`<Execute|command>`，上层应用调用该接口请求每个进程执行命令 `command`；另一个是输出事件`<Output|result>`，表示状态机执行完命令`command`后且输出了结果`result`。

RSM1一致特性要求每个正确的进程以相同的顺序输出结果，但并不保证"先请求、先执行"。RSM2终止特性要求状态机在执行每个命令时是无阻塞的。

## 结构模型

复制状态机由共识模块、日志模块、状态机组成。通过共识模块保证各个节点日志的一致性，然后各个节点基于同样的日志、顺序执行指令，最终各个复制状态机的结果实现一致。

在基于复制状态机实现的分布式存储系统中，Raft等一致性算法只能确保各个节点的日志一致性。而对于server从日志里面获取已提交的日志条目，将其应用到状态机的过程，跟Raft算法本身无关，属于server本身的数据存储逻辑。

# 从Paxos到Raft

Paxos协议是一个著名的共识算法，由Lamport在1989年首次提出。在Basic Paxos中，有提议者（Proposer）、接受者（Acceptor）、学习者（Learner）三种角色。Basic Paxos是通过二阶段提交的方式来达成共识的，分为准备（Prepare）阶段和接受（Accept）阶段。

为了实现一系列值的共识，Multi-Paxos思想引入领导者和优化Basic Paxos执行来解决。采用"当领导者处于稳定状态时，省掉准备阶段，直接进入接受阶段"这个优化机制。

不幸的是，Paxos有两个显著的劣势。第一个劣势是Paxos非常难以理解。第二个问题是它没有为实用的实现提供良好的基础。

因为这些问题，Diego Ongaro和John Ousterhout于2014年设计出一个能替代Paxos的且比Paxos的属性更好的共识算法——Raft协议。

# Raft协议概览

Raft是一个用来管理多副本日志（replicated log）的共识算法。Raft通过问题分解，将复杂的共识问题拆分成三个子问题：

- Leader选举（Leader election），Leader故障后集群能快速选出新Leader；
- 日志复制（Log replication），集群只有Leader能写入日志，Leader负责复制日志到Follower节点；
- 安全性（Safety Property），一个任期内集群只能产生一个Leader、已提交的日志条目在发生Leader选举时，一定会存在更高任期的新Leader日志中。

## 成员身份

Raft算法支持Leader、Follower和Candidate 3种状态。Raft算法是强领导者模型，集群中只能有一个Leader。

## 领导者选举

Raft将时间划分为任意长度的term（任期），term被编号为连续的整数。每个term从选举开始，在选举中一个或多个candidate会试图成为leader。Raft使用了随机选举超时（election timeout）时间特性以确保投票决裂很少发生。

选举流程：节点超时→增加任期编号→推举自己为候选人→给自己投票→向其他节点发送RequestVote RPC→赢得大多数选票→成为领导者→周期性发送心跳消息。

## 日志复制

日志项（Log entry）包含指令（Command）、索引值（Log index）、任期编号（Term）。Raft的日志复制是一个优化后的二阶段提交（将二阶段优化成了一阶段），减少了一半的往返消息。

当leader知道了一条log entry已经被复制到大多数服务器上，那么该条目会变成committed的。这还会提交在leader的日志中所有之前的条目。

在实际环境中，Raft通过以领导者的日志为准，来实现各节点日志的一致。领导者通过日志复制RPC的一致性检查，找到跟随者节点上与自己相同日志项的最大索引值，然后强制跟随者更新覆盖的不一致日志项。

## 安全性

Raft在所有时刻都保证了State Machine Safety Property：如果任意服务器将一个特定的日志条目应用到了其状态机中，那么不会有服务器apply具有相同index的不同Log Entry。

### 选举约束

在一次选举中，每一个服务器节点最多会对一个任期编号投出一张选票，按照"先来先服务"的原则进行投票。日志完整性高的跟随者拒绝投票给日志完整性低的候选人。

Raft通过比较日志最后一个条目的index和term来确定哪个日志更新。leader永远不会覆写它的日志中已存在的条目。

### 对提交的限制与幽灵复现问题

Leader只允许commit包含当前term的日志。Raft算法要求Leader当选后立即追加一条no-op的特殊内部日志，并立即同步到其它节点，实现前面未Committed日志全部隐式提交。

# 日志压缩

Raft提供了一种机制去清除日志里积累的陈旧信息，叫做日志压缩（Log Compaction）。快照（Snapshot）是一种常用的、简单的日志压缩方式。

每个节点可以独立的选择何时做snapshot，而不是由leader统一发起snapshot。同步快照使用一个新的RPC方法，叫做InstallSnapshot RPC。

# 集群成员变更

Raft论文中给出了一种无需停机的、自动化的改变集群成员的方式，将集群成员配置作为一个特殊日志从leader节点同步到其它节点去。

最初实现成员变更的是联合共识（Joint Consensus），后来Raft的作者提出了一种改进后的方法：单节点变更（single-server changes）。该算法每次只允许增加或移除一个节点，只有当上一轮成员变更结束，才能开始下一轮。

etcd/raft中的配置的应用时间与论文中的不同。etcd/raft在"apply-time"应用新配置的方式，可以保证配置在应用前已被提交，因此不需要论文中提到的回滚旧配置的操作。
